Fase 2: Infraestrutura e Base do Projeto - Projeto ChamaEste documento define os padrões rígidos para a inicialização dos repositórios, orquestração de contentores e configuração base do backend e frontend. A não conformidade nesta fase resultará em refatoração severa nas fases posteriores.1. Orquestração e Ambientes (Docker)É estritamente proibido o uso de instalações locais de bases de dados (como SQLite ou PostgreSQL nativo) para o desenvolvimento do backend. A paridade de ambiente é inegociável.1.1 docker-compose.yml (Backend)O ambiente local deve subir com um único comando (docker-compose up -d).Serviço db: Imagem oficial do postgres:15-alpine.Requisito: Mapeamento de volume (./data/db:/var/lib/postgresql/data) para garantir persistência local entre reboots.Serviço redis: Imagem oficial do redis:7-alpine.Justificativa: Dependência obrigatória para o Channel Layer do Django Channels (WebSockets).Serviço web (Django): Construído a partir de um Dockerfile baseado em python:3.11-slim.Requisito: Variáveis de ambiente injetadas via ficheiro .env (nunca hardcoded). Dependência estrita do serviço db (usar script wait-for-it ou configuração de healthcheck do compose).2. Setup Base do Backend (Django + DRF)2.1 Modelo de Usuário Customizado (CRÍTICO)Antes de executar a primeira migração (python manage.py migrate), o CustomUser deve estar definido.Implementação: Herdar de AbstractBaseUser e PermissionsMixin. Definir email como USERNAME_FIELD.Justificativa: Tentar trocar o modelo de utilizador base do Django a meio do projeto obriga, na maioria dos casos, a destruir a base de dados inteira.2.2 Estrutura de DiretóriosO projeto deve abandonar a estrutura padrão plana do Django e adotar modularidade:config/: (Substitui a pasta com o nome do projeto) Contém settings.py, urls.py, wsgi.py, asgi.py.apps/: Diretório base para as aplicações (ex: apps/users, apps/organizations, apps/tickets).Justificativa: Previne a poluição do diretório raiz (root) e isola o domínio do negócio.2.3 Preparação ASGIO ficheiro asgi.py deve ser configurado imediatamente, encapsulando a aplicação Django com o ProtocolTypeRouter do Channels. O servidor local deve ser o daphne ou uvicorn, e não o servidor de desenvolvimento síncrono padrão do Django.3. Setup Base do Frontend (Ionic + React)A arquitetura exige separação de repositórios (Multi-repo) ou configuração estrita de Workspaces (Monorepo). Não misturar código Python e JavaScript no mesmo diretório de aplicação.3.1 Inicialização e CapacitoresComando base: ionic start chama-app blank --type=react --capacitor.Requisito: O Capacitor deve estar configurado desde o dia 1 para garantir que bibliotecas incompatíveis com o ambiente nativo mobile não sejam introduzidas acidentalmente.3.2 Gestão de Variáveis de AmbienteImplementação de ficheiros .env.development e .env.production.Justificativa: URLs de API (ex: http://localhost:8000/api/ vs https://api.chama.com/) não podem estar fixas no código fonte.4. Controlo de Qualidade Inicial (Shift-Left)A qualidade do código não é opcional e deve ser automatizada antes dos commits.4.1 Pre-commit Hooks (Python)Instalação obrigatória do pacote pre-commit no ambiente virtual local.Black: Formatador opinativo. Sem debates sobre estilo de código.Flake8: Linter de estilo e deteção de erros lógicos não capturados pelo Black.isort: Ordenação alfabética e por tipo de blocos de imports.Justificativa: O código não formatado será rejeitado localmente pelo hook do git antes de poluir o repositório.5. Critérios de Validação (Gateways para a Fase 3)A Fase 2 só é considerada concluída quando as seguintes métricas forem validadas na máquina de desenvolvimento:Validação de Orquestração: O comando docker-compose up -d sobe o Postgres, Redis e Django simultaneamente sem erros (exit code 0).Validação de Migração Zero: O painel de administração do Django abre em localhost, e a tabela de utilizadores reflete o modelo base focado em email (ausência do campo genérico username).Validação de Linter: Tentar fazer commit de um ficheiro Python mal formatado resulta num bloqueio automático do terminal pelo pre-commit.Conectividade Frontend: A aplicação Ionic corre em navegador (porta padrão 8100) e executa um GET ou requisição de healthcheck básica com sucesso contra a API do Django no contentor.
