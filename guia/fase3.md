Fase 3: Desenvolvimento do Core Backend - Projeto ChamaEste documento estabelece as regras de engenharia para a implementação da API RESTful, modelos de dados, máquina de estados e comunicação em tempo real (WebSockets).1. Implementação do ORM e Camada de DadosA tradução do modelo físico (DER da Fase 1) para o Django ORM deve focar na integridade e performance de leitura.1.1 Modelos e HierarquiaTreebeard: A entidade Department deve obrigatoriamente herdar de MP_Node. A manipulação da árvore (adição/movimentação de nós) deve usar exclusivamente a API do django-treebeard (add_child(), add_sibling()). Edição manual dos campos path, depth ou numchild é estritamente proibida.Índices Estratégicos: Adicionar db_index=True em todos os campos de filtragem frequente (status, organization_id, assignee_id).Deleção Lógica (Soft Delete): (Recomendação forte) Para entidades transacionais (Ticket, AuditLog), implementar um manager que filtra is_deleted=False em vez de apagar os registos (evita perda de histórico de auditoria).2. API REST (Django REST Framework)2.1 Autenticação e Permissões GranularesInstalar e configurar djangorestframework-simplejwt. O tempo de expiração do access token não deve exceder 15 minutos.Criar classes de permissão customizadas no DRF (ex: IsSupervisor, IsExecutor). A lógica de permissão deve aceder ao user.membership para validar o contexto hierárquico antes de permitir operações destrutivas.2.2 Serializers (Read/Write Separation)Proibido usar o mesmo Serializer para Entrada e Saída em entidades complexas.ReadSerializer: Focado em performance e relações aninhadas (ex: retornar os dados do creator e assignee).WriteSerializer: Aceita apenas IDs e primitives, validando a entrada. Protege contra vulnerabilidades de atribuição em massa (Mass Assignment).Lógica de negócio é proibida dentro do método .create() ou .update() do Serializer. Delegação para uma camada de Services.2.3 Prevenção de Problemas N+1Endpoints de listagem (GET /tickets/) devem obrigatoriamente utilizar .select_related() para chaves estrangeiras únicas (ex: creator, department) e .prefetch_related() para relações múltiplas (ex: comments, tags).Auditoria: O pacote django-debug-toolbar (ou inspeção de query de teste) deve provar que listar 100 chamados gera o mesmo número de queries SQL que listar 1 chamado.3. Máquina de Estados e Camada de ServiçoA lógica de transição não pertence à API (View). Pertence ao núcleo de negócio.3.1 Serviço de Transição (services.py)Criar uma função isolada para gerir a mudança de estados:def transition_ticket(ticket, actor, action):Validar a transição de acordo com a matriz definida na Fase 1 (ex: Executor não pode aprovar ticket Em Revisão).Gravar o estado atualizado no Ticket e gerar um registo em TicketTransitionLog dentro da mesma transação de base de dados (with transaction.atomic():).Disparar um sinal assíncrono (ou Celery/Channels) para notificação após o commit bem-sucedido.4. WebSockets (Django Channels)O frontend precisa de atualizações de interface sem efetuar polling HTTP.4.1 Consumidores AssíncronosCriar um AsyncWebsocketConsumer. O consumidor síncrono bloqueia o event loop e consome recursos desnecessários.Agrupamento (Channel Layers): Quando um utilizador faz a conexão, deve ser subscrito num grupo específico (ex: user_uuid ou department_uuid).Segurança no WS: A conexão inicial WebSocket DEVE validar o JWT passado na query string ou cabecalho de conexão antes de aceitar (self.accept()). Conexões não autenticadas devem ser rejeitadas instantaneamente.5. Critérios de Validação (Gateways para a Fase 4)A Fase 3 só é considerada concluída e o trabalho de Frontend (Fase 4) só pode começar quando:Documentação Automática: O endpoint de documentação (via drf-spectacular) está online (/api/docs/) listando todos os endpoints, métodos permitidos e esquemas JSON (Request/Response). O frontend não deve adivinhar payloads.Cobertura da FSM: Existem testes unitários cobrindo 100% da máquina de estados. Tentar executar uma transição inválida (ex: Criador tentando fechar chamado) resulta em erro HTTP 403 ou 400 documentado, comprovado por código.Eficiência de Base de Dados: Consultas ao endpoint de chamados /api/tickets/ não exibem anomalias N+1 no query log.Validação WebSocket: Um cliente de teste local (ex: wscat ou Postman) consegue conectar via ws://, autenticar, e recebe um payload JSON automaticamente quando o estado de um chamado muda no painel de administração do Django.
