Fase 1: Fundação e Modelagem Arquitetural - Projeto ChamaEste documento define o padrão de engenharia para a modelagem relacional, contratos de API e prototipação de baixa fidelidade, estabelecendo os requisitos rígidos para validação antes da transição para a Fase 2 (Infraestrutura).1. Modelagem Relacional (DER / Modelo Físico PostgreSQL)A modelagem prioriza integridade referencial, normalização até a 3FN para dados transacionais e otimização para leitura hierárquica usando Materialized Path (django-treebeard).1.1 Entidades CoreUser (Custom AbstractBaseUser)id (UUID4, PK)email (VARCHAR(255), UNIQUE, INDEX)password_hash (VARCHAR)is_active (BOOLEAN, DEFAULT TRUE)created_at (TIMESTAMPTZ)Justificativa: UUID previne enumeração de usuários (IDOR). E-mail como identificador primário é padrão de mercado, dispensando o username do Django padrão.Organizationid (UUID4, PK)name (VARCHAR(100), INDEX)slug (VARCHAR(100), UNIQUE, INDEX)logo_url (VARCHAR(2048), NULL)created_at (TIMESTAMPTZ)Justificativa: O slug (ex: empresa-acme) permite roteamento amigável e multitenancy na URL, facilitando a separação lógica no frontend.Department (Nó da Árvore - via django-treebeard MP_Node)id (UUID4, PK)organization_id (UUID4, FK -> Organization, ON DELETE CASCADE, INDEX)name (VARCHAR(100))path (VARCHAR(255), UNIQUE, INDEX) -- Gerenciado pelo treebearddepth (INT) -- Gerenciado pelo treebeardnumchild (INT) -- Gerenciado pelo treebeardJustificativa: Delegação CASCADE garante que a destruição de uma organização limpe seus subdepartamentos. O uso de path resolve a recursão de consultas hierárquicas em $O(1)$ ou $O(\log n)$.Role (Papéis Predefinidos na Aplicação)id (INT, PK)name (VARCHAR(50), UNIQUE) -- Valores estritos: 'Criador', 'Executor', 'Supervisor', 'Admin'Membership (Junção Multitenant)id (UUID4, PK)user_id (UUID4, FK -> User, ON DELETE CASCADE, INDEX)organization_id (UUID4, FK -> Organization, ON DELETE CASCADE, INDEX)department_id (UUID4, FK -> Department, NULL, ON DELETE SET NULL)role_id (INT, FK -> Role)Constraint: UNIQUE(user_id, organization_id)Justificativa: Um usuário pertence a uma organização sob um papel principal, associado opcionalmente a um departamento específico (ex: um 'Admin' pode não estar em um departamento, mas um 'Executor' deve estar).1.2 Entidades Transacionais (Máquina de Estados)Ticket (Chamado)id (UUID4, PK)organization_id (UUID4, FK -> Organization, ON DELETE CASCADE, INDEX)department_id (UUID4, FK -> Department, ON DELETE RESTRICT, INDEX)creator_id (UUID4, FK -> User, ON DELETE RESTRICT)assignee_id (UUID4, FK -> User, NULL, ON DELETE SET NULL, INDEX)title (VARCHAR(200))description (TEXT)status (VARCHAR(20), INDEX) -- Enum: ABERTO, ANDAMENTO, REVISAO, CONCLUIDO, CANCELADOpriority (INT) -- 1(Baixa), 2(Média), 3(Alta), 4(Crítica)due_date (TIMESTAMPTZ, NULL, INDEX)created_at (TIMESTAMPTZ)updated_at (TIMESTAMPTZ)Justificativa: organization_id redundante aqui é necessário para consultas de multitenancy rápidas sem precisar realizar JOIN até o Department. ON DELETE RESTRICT no department_id e creator_id impede a quebra do histórico financeiro/auditoria se um usuário ou departamento for removido (Soft Delete seria a evolução aqui, mas adiciona complexidade desnecessária agora).TicketTransitionLog (Auditoria da Máquina de Estados)id (UUID4, PK)ticket_id (UUID4, FK -> Ticket, ON DELETE CASCADE, INDEX)actor_id (UUID4, FK -> User, NULL, ON DELETE SET NULL)from_status (VARCHAR(20))to_status (VARCHAR(20))transitioned_at (TIMESTAMPTZ, DEFAULT NOW(), INDEX)Justificativa: Tabela apend-only. Fundamental para cálculo de métricas (SLA, tempo médio de atendimento, gargalos). Substitui a falta de histórico do WhatsApp.2. Contratos de API (Padrão OpenAPI/RESTful)A API deve ser estritamente baseada em recursos (Resource-Oriented) e utilizar HTTP Verbs corretamente. Respostas em JSON.2.1 Autenticação (/api/auth/)POST /api/auth/token/: Recebe email e password. Retorna access (JWT) e refresh.POST /api/auth/token/refresh/: Renova o access token.2.2 Organizações e Hierarquia (/api/organizations/)POST /api/organizations/: Cria org (requer auth).GET /api/organizations/{slug}/: Detalhes da org (verifica Membership).GET /api/organizations/{slug}/departments/: Retorna a árvore hierárquica.Payload Esperado: Resposta aninhada ou lista linear com IDs de subordinação (depende da escolha de otimização de renderização no Ionic). Padrão recomendado: Lista linear com parent_id para montagem no cliente.2.3 Gestão de Chamados (/api/tickets/)GET /api/tickets/: Listagem com filtros pesados via query params (ex: ?org=slug&status=ABERTO&assignee=uuid).POST /api/tickets/: Criação.Payload: { "org_slug": "...", "department_id": "...", "title": "...", "description": "..." }GET /api/tickets/{id}/: Detalhes completos.PATCH /api/tickets/{id}/: Edição parcial (descrição, prioridade). Não altera status.POST /api/tickets/{id}/transitions/: Endpoint Crítico da FSM.Payload: { "action": "iniciar" | "pausar" | "enviar_revisao" | "aprovar" | "rejeitar" | "cancelar" }Justificativa: Status não é editável diretamente via PATCH para evitar inconsistências. Transições de estado são eventos (CQRS leve), validados no backend de acordo com a regra de Role (quem chama) vs. State (estado atual).3. Critérios de Validação (Gateways para a Fase 2)A Fase 1 só é considerada concluída quando as seguintes afirmações forem comprovadas:Validação de Multitenancy: O DER garante matematicamente que os dados da Organização A (tickets, departamentos) não vazarão para a Organização B sob nenhuma circunstância de consulta base (garantido pelas FKs e o organization_id replicado estrategicamente).Validação de Concorrência de FSM: O contrato de API (/transitions/) deixa claro que a mudança de estado não é uma simples alteração de string (PATCH), exigindo uma transação no banco de dados (ACID) que crie o log de auditoria no mesmo commit.Prototipagem Validada: O diagrama de telas (que deve ser feito no Figma ou similar) contempla, no mínimo, três fluxos fechados:Login/Seleção de Organização.Painel do Supervisor (Visão da fila do Departamento).Painel do Executor (Visão do Cronograma/Kanban diário).
