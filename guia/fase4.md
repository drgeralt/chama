Fase 4: Desenvolvimento Frontend e Mobile - Projeto ChamaEste documento estabelece as diretrizes arquiteturais para a aplicação cliente construída com Ionic e React. O foco é a resiliência offline-first, gestão de estado eficiente e integração nativa via Capacitor.1. Gestão de Estado e Cache de DadosA utilização exclusiva da Context API do React para dados assíncronos é proibida. Causa degradação de performance por renderizações excessivas (re-render hell).1.1 Estado de Servidor (Server State)Ferramenta Obrigatória: TanStack Query (React Query) ou Redux Toolkit Query (RTK Query).Justificativa: Os dados dos chamados pertencem ao servidor, não ao cliente. Estas bibliotecas gerem cache, invalidação automática, retries e estados de carregamento (loading/error) sem poluir a árvore de componentes.1.2 Estado de Interface (UI State)Ferramenta Obrigatória: Zustand (preferencial por ser minimalista) ou Redux.Justificativa: Gerir qual aba está ativa, estado de modais ou o tema (Dark/Light mode). O estado deve ser atómico e não acoplado à hierarquia de componentes.2. Comunicação HTTP e Segurança (JWT)O cliente deve ser robusto perante falhas de rede e expiração de sessões.2.1 Intercetores HTTP (Axios)Criar uma instância global do Axios (api.ts).Regra de Refresh: Implementar um interceptor de resposta que capture erros 401 Unauthorized. Ao capturar, a aplicação deve pausar as requisições, efetuar a chamada para /api/auth/token/refresh/, atualizar o access token em armazenamento seguro, e repetir a requisição original de forma transparente para o utilizador.2.2 Armazenamento SeguroWeb: Utilizar localStorage ou sessionStorage (com as devidas mitigações de XSS no código).Mobile (Capacitor): É obrigatório o uso do plugin @capacitor/preferences para armazenar o Refresh Token. Nunca armazenar tokens em texto plano se o dispositivo permitir extração via root/jailbreak.3. Comunicação em Tempo Real (WebSockets)O Ionic não deve fazer polling (requisições repetidas GET /tickets/) para verificar se há novos chamados.3.1 Gestão da Conexão WSUtilizar bibliotecas resilientes como react-use-websocket ou implementar lógica estrita de Exponential Backoff (tentativas de reconexão com intervalos crescentes caso a rede caia).A conexão inicial do WebSocket deve enviar o JWT para validação, conforme exigido na Fase 3.3.2 Sincronização de CacheQuando o WebSocket receber um evento (ex: {"type": "ticket_updated", "data": {...}}), o frontend não deve fazer um novo GET à API.Regra: O payload recebido no WebSocket deve atualizar diretamente o cache local da biblioteca de estado de servidor (ex: queryClient.setQueryData() no React Query), refletindo a mudança na interface instantaneamente (Optimistic UI).4. Integração Nativa Mobile (Capacitor)WebSockets morrem quando a aplicação móvel é minimizada ou o ecrã é bloqueado (restrição dos sistemas operativos Android/iOS para poupar bateria).4.1 Notificações Push (Background)Implementação Crítica: É obrigatório integrar o Firebase Cloud Messaging (FCM) através do plugin @capacitor/push-notifications.Justificativa: Se o utilizador é um Executor, ele precisa de saber que recebeu um novo chamado mesmo com o telemóvel no bolso. O backend (Django) enviará o push notification via FCM, que o sistema operativo recebe e exibe, redirecionando para a aplicação.5. Critérios de Validação (Gateways para a Fase 5)A Fase 4 só é considerada concluída quando as seguintes afirmações forem comprovadas num dispositivo real ou emulador nativo (não apenas no browser):Renovação de Sessão Automática: Um access token expirado não obriga o utilizador a fazer login novamente. O sistema renova o token automaticamente em plano de fundo através do refresh token.Atualização Reativa (Zero-Reload): Alterar o estado de um chamado no painel de administração do Django atualiza o ecrã do telemóvel em menos de 500ms através da conexão WebSocket, sem que o utilizador precise recarregar a lista.Persistência Mobile: Fechar a aplicação Ionic (arrastar para cima) e voltar a abrir mantém o utilizador autenticado (verificação do @capacitor/preferences).Notificação de Fundo: Com a aplicação minimizada, a criação de um chamado crítico dispara uma notificação do sistema operativo no emulador/dispositivo Android.
