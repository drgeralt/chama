Fase 6: Deploy, CI/CD e Apresentação - Projeto ChamaEste documento define os padrões para a colocação do sistema em ambiente de produção. O objetivo é garantir previsibilidade, alta disponibilidade durante a avaliação e eliminar intervenções manuais no servidor.1. Infraestrutura de Backend e Base de Dados (PaaS)O uso de Infraestrutura como Serviço (IaaS, ex: AWS EC2 vazia) é um desperdício de tempo para esta escala, a não ser que seja requisito da disciplina. A abordagem correta é Plataforma como Serviço (PaaS).1.1 Plataforma Alvo (Render ou Railway)Serviços Exigidos: O ambiente de produção necessita de três instâncias separadas:Base de Dados PostgreSQL (Gerida).Instância Redis (Para o Django Channels/WebSockets).Serviço Web (Django executado via Dockerfile).Justificativa: Contentores são efémeros. Qualquer ficheiro gravado no disco do Serviço Web desaparecerá no próximo deploy. O estado deve viver estritamente no PostgreSQL e Redis.1.2 Gestão de Ficheiros (Estáticos e Media)Servir ficheiros via Gunicorn/Daphne em produção é ineficiente e arriscado.Ficheiros Estáticos (CSS/JS do Admin): Utilizar a biblioteca WhiteNoise. O Django compila os estáticos (collectstatic) e o WhiteNoise serve-os diretamente com compressão Brotli/Gzip.Ficheiros de Media (Uploads, Logos de Organizações): Integração obrigatória com AWS S3 ou Cloudinary via django-storages.Justificativa: Como o contentor Web é destruído a cada deploy, guardar o logótipo de uma Organização no disco local resultará em perda de dados (Broken Links).2. Infraestrutura de Frontend WebA aplicação web deve ser desacoplada do backend. O Django não deve renderizar templates HTML para a aplicação Ionic/React.2.1 Hospedagem de Estáticos (Vercel ou Netlify)Configurar o repositório do frontend para deploy automático na Vercel/Netlify.Variáveis de Ambiente: Garantir que o REACT_APP_API_URL (ou equivalente no Vite) aponta para o domínio público do backend (ex: https://api.chama-app.com).Rewrite Rules: Configurar o servidor (Vercel vercel.json ou Netlify _redirects) para redirecionar todas as rotas para o index.html (/* /index.html 200), delegando o roteamento para o React Router.3. Integração Contínua e Entrega Contínua (CI/CD)Deploys manuais são proibidos. A ponte entre o código local e a produção deve ser automatizada.3.1 GitHub ActionsImplementar um ficheiro YAML (.github/workflows/deploy.yml) com o seguinte pipeline:Linting: Executar flake8 e black --check.Testing: Subir um PostgreSQL de teste no GitHub Actions, aplicar migrações e correr pytest.Build & Push: Apenas se a etapa de testes passar, construir a imagem Docker e enviá-la para a plataforma de PaaS (ou notificar o webhook do Render/Railway para iniciar o build).Justificativa: Um commit com falhas lógicas nunca chegará a produção, protegendo o ambiente de apresentação.4. Build Mobile (Android/APK)A demonstração requer a aplicação a correr nativamente para comprovar a eficácia das Notificações Push e dos WebSockets em background.4.1 Empacotamento via CapacitorCompilar o projeto de produção: npm run build (gera os binários web miniaturizados).Sincronizar o Capacitor: npx cap sync android.Geração do Binário: Abrir o projeto no Android Studio e gerar o APK assinado (Build > Generate Signed Bundle / APK).Restrição: O APK final deve ser testado num dispositivo físico antes da banca. Emuladores mascaram problemas de rede e CORS.5. Critérios de Validação (Gateways Finais)A Fase 6 e o projeto só são considerados finalizados quando as seguintes condições forem inquestionavelmente provadas:Força HTTPS: É impossível aceder à API ou ao Frontend via http://. Todos os acessos são redirecionados automaticamente para https:// (mitigação de ataque Man-in-the-Middle).Separação de Domínios: O Frontend corre num domínio independente do Backend, e o CORS está estritamente configurado para permitir a comunicação apenas entre estes nós.Persistência Volátil Comprovada: Fazer um deploy forçado do backend (reiniciar a máquina no Render) não apaga os logótipos das Organizações, provando que o S3/Cloudinary está a operar corretamente.Instalável Nativo: Existe um ficheiro .apk funcional que pode ser enviado para a banca avaliadora, que comunica corretamente com o backend em produção sem falhas de certificado.
