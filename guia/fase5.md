Fase 5: Testes, Monitorização e Refinamento - Projeto ChamaEste documento define os padrões para a garantia de qualidade (QA) e observabilidade do sistema. O objetivo não é atingir 100% de cobertura de código (code coverage), mas garantir zero falhas nos fluxos críticos de negócio e capturar exceções silenciosas antes do lançamento.1. Estratégia de Testes (Backend)A utilização do pytest e pytest-django é obrigatória. O módulo nativo unittest do Python é verboso e desencorajado.1.1 Geração de Dados (Fixtures)Ferramenta Obrigatória: factory_boy.Justificativa: Criar dados de teste manualmente com o ORM (ex: criar Organização, depois Departamento, depois Utilizador, depois Ticket) torna os testes frágeis e difíceis de ler. O factory_boy permite criar instâncias complexas com um comando (ex: TicketFactory(status='ABERTO')).1.2 Foco dos Testes Unitários e IntegraçãoMáquina de Estados (FSM): Testar exaustivamente o services.py. Deve ser provado via código que uma transição ilegal (ex: Executor forçar estado para Concluído) levanta a exceção correta (ValidationError ou PermissionDenied).Isolamento de Multitenancy: Criar dois utilizadores em Organizações diferentes. O teste de integração no endpoint GET /api/tickets/ tem de falhar explicitamente se o Utilizador A conseguir ver o Ticket da Organização B.2. Estratégia de Testes (Frontend/E2E)Testes de componentes UI (ex: testar se um botão é azul) são um desperdício de tempo nesta fase. O foco é End-to-End (E2E).2.1 Testes de Caminho CríticoFerramenta Recomendada: Playwright (superior ao Cypress para lidar com múltiplos contextos de navegador e WebSockets).Cenários Obrigatórios (E2E):Autenticação com sucesso e falha.Criação de um chamado (Utilizador A) e verificação do aparecimento no Backlog.Transição de estado: Utilizador B (Executor) move o chamado para "Em Andamento".3. Observabilidade e Tratamento de ErrosO projeto não pode depender de utilizadores ou professores a reportarem que "o ecrã ficou branco" ou "deu erro 500".3.1 Captura de Exceções (Sentry)Configurar projetos separados no Sentry.io para o Django (Backend) e para o React (Frontend).Frontend: Configurar Source Maps no processo de build. Um erro de JavaScript minimizado não tem utilidade para depuração.Backend: O Sentry deve estar configurado para capturar o utilizador autenticado (user.id) em cada exceção reportada, permitindo rastrear quem sofreu a falha.3.2 Auditoria de Performance (Pre-Deploy)Utilizar o django-debug-toolbar em ambiente local para inspecionar os tempos de resposta.Qualquer requisição à API (excluindo geração de relatórios complexos) deve ter um tempo de processamento no servidor inferior a 200ms e executar menos de 10 queries SQL.4. Segurança e Refinamento FinalCORS (Cross-Origin Resource Sharing): O backend deve rejeitar requisições de origens não reconhecidas. Apenas os domínios de produção e de desenvolvimento local (localhost:8100) devem estar na CORS_ALLOWED_ORIGINS.Exposição de Dados Sensíveis: Garantir que o ReadSerializer do Utilizador não expõe a password_hash ou campos de reset de palavra-passe.5. Critérios de Validação (Gateways para a Fase 6 - Deploy)A Fase 5 só é considerada concluída e o projeto está pronto para ir para produção quando as seguintes condições forem satisfeitas:Pipeline Verde: Todos os testes do pytest passam com sucesso. A cobertura mínima exigida na camada de Serviços (FSM) é de 100%.Sentry Integrado: Uma exceção forçada manualmente no frontend (ex: throw new Error('Teste')) e no backend (ex: divisão por zero numa view) aparece no painel de controlo do Sentry com o respetivo stack trace.Bloqueio Multitenant: Existe pelo menos um teste automatizado a passar que comprova que é impossível aceder a dados cruzados entre organizações.Sem N+1 Detetados: A visualização de uma lista com 50 chamados não dispara 50 relatórios de queries SQL no Sentry ou no Django Debug Toolbar.
